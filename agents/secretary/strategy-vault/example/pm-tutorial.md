# PM Tutorial — Building Context Intelligence Under a Zero-Egress Constraint

*Derived from commit `a1b2c3d`: feat: PDF content extraction for identity enrichment*

---

## The Design Pattern: Constraint-Driven Signal Minimisation

This tutorial extracts a reusable PM pattern from Aura's content extraction implementation.

**The pattern**: When you cannot store raw user data, force your team to define the minimum viable signal before writing a single line of code.

---

## Step 1 — Define the Privacy Boundary First

Before scoping the feature, write the invariant:

> *Raw [content type] is never written to disk. Only derived [signal type] survives.*

For Aura's PDF extraction:
> *Raw PDF text is never written to disk. Only keyword tokens survive.*

This sentence becomes your acceptance criterion for every implementation PR. If a PR violates it, it doesn't ship.

---

## Step 2 — Work Backwards from the Signal, Not the Data

Ask: "What is the minimum representation of [content] that answers our question?"

For identity enrichment from PDFs:
- Question: "What domain is this person focused on right now?"
- Minimum representation: top 5 keyword tokens from a domain lexicon
- Not needed: full text, embeddings, summaries, entity lists

The minimum representation is almost always smaller than what a cloud-first system would collect. This is the product benefit of the constraint — it forces specificity.

---

## Step 3 — Design the Discard Point Explicitly

The discard isn't an implementation detail — it's a product requirement. Make it visible:

```
Extract → Map → **DISCARD** → Persist tokens only
```

In code review, verify the discard point is architecturally enforced (function returns tokens, not text) rather than procedurally enforced (a comment saying "don't use this variable").

---

## Step 4 — Set a Coverage Baseline Before Shipping

Measure lexicon coverage on a representative sample *before* the feature launches. For content extraction:

| Lexicon size | Coverage on test set | Avg. confidence |
|-------------|---------------------|-----------------|
| 50 tokens | 61% | 52% |
| 200 tokens | 87% | 78% |
| 500 tokens | 94% | 89% |

Ship at the coverage level that meets your accuracy hypothesis. Don't over-engineer upfront.

---

## Applying This Pattern to Your Domain

Replace "PDF" with your content type and "keyword tokens" with your signal type:

- Legal documents → clause type tags
- Medical records → condition category flags
- Research papers → citation domain labels
- Real estate listings → property attribute tokens

The pattern holds. The privacy constraint holds. The product benefit — forced signal minimisation — holds.
